### 1.1 一句话版本

`weapp-vite` 的核心价值，不是把小程序“框架化”，而是把原生小程序研发“现代化”。

### 1.2 开场要先解决的认知偏差

很多听众一听到名字，会先把 `weapp-vite` 理解成“给小程序换了一个 Vite 构建器”。这不是错，但明显不完整。更准确的说法是：

- 它保留原生小程序的目录结构、语法习惯和运行时边界。
- 它补齐现代前端工程化能力。
- 它把分包、npm、组件治理、IDE 调试、截图验收、AI 协作串成完整研发链路。

## 2. 叙事主线

### 2.1 核心信息

这场分享的核心信息只有一句：

`weapp-vite` 不是让原生团队推倒重来，而是在保留原生模型的前提下，把现代工程化、复杂场景治理、调试闭环和 AI 协作一次性补齐。

### 2.2 对比主线

| 现在的大多数原生小程序项目                    | 使用 weapp-vite 之后                                       |
| --------------------------------------------- | ---------------------------------------------------------- |
| 还能做，但工程化体验落后                      | 仍然写原生，但开发体验更接近现代前端工程                   |
| 页面、组件、分包、npm 依赖维护成本高          | 自动路由、自动导入、分包策略和 npm 落位更可控              |
| 调试链路割裂，编辑器、终端、DevTools 来回切换 | `dev/build/analyze/open/logs/screenshot/compare` 串成闭环  |
| AI 只能解释代码，难以真正参与小程序验收       | MCP、`AGENTS.md`、本地文档、日志与截图让 AI 能执行真实动作 |

### 2.3 三个支撑论点

1. `weapp-vite` 优先解决的是“原生团队不想换范式，但必须升级研发效率”的矛盾。
2. 它的优势在复杂项目里更明显，尤其是分包、npm、组件数量多、多人协作、需要自动化验收的项目。
3. 它已经不只是构建工具，而是小程序研发工具链，包括脚手架、CLI、IDE 自动化、AI 接入和运行时配套能力，其中 `wv prepare` 与 `layout` 明显吸收了 Nuxt 的工程化思想，再按小程序场景做了改造。

### 2.4 可以点出的设计借鉴

如果你想让这场分享更有“技术判断力”，建议明确说出 `weapp-vite` 不是凭空长出这些能力，而是有意识地借鉴了成熟前端框架的设计思想，其中最值得点名的是 Nuxt：

- `wv prepare` 的思路接近 `nuxt prepare`。它把 `.weapp-vite/*` 这一批受管支持文件提前生成出来，让类型检查、编辑器提示、安装后初始化和老项目渐进接入更稳定。
- `layout` 的思路接近 Nuxt 的 `app/layouts`。它把“页面壳”从单页实现里抽出来，用 `src/layouts` 和 `definePageMeta({ layout })` 组织默认布局和命名布局，让页面结构复用更自然。
- 但 `weapp-vite` 不是把 Nuxt 直接搬过来，而是把这些思想重新落到了小程序语法、页面模型和构建产物约束里。

讲稿备注：

## 5. 可直接念的开场与收尾

### 5.1 开场白

大家今天看到 `weapp-vite` 这个名字，第一反应可能是：它是不是就是一个给小程序换成 Vite 的构建工具？这个理解只对了一半。更准确地说，`weapp-vite` 想做的是一件更现实的事：在不推翻原生小程序研发方式的前提下，把现代前端工程化、复杂项目治理、调试验收闭环和 AI 协作能力带进来。它不是让团队换一种技术信仰，而是帮助原生团队用更低成本完成一次研发方式升级。

### 5.2 收尾

如果要用一句话总结 `weapp-vite` 的价值，我建议这样说：它不是把小程序框架化，而是把原生小程序研发现代化。对于那些仍然重视原生能力、又已经明显感受到工程化和协作成本上升的团队来说，它提供的是一条更现实、也更容易落地的升级路径。

## 6. 现场 Demo 建议

### 6.1 推荐 Demo 路线

如果你现场有 3 到 4 分钟演示时间，推荐只演一条完整链路，不要贪多：

1. 打开一个 `weapp-vite` 项目。
2. 展示页面目录和组件目录。
3. 说明自动路由与自动导入组件如何减少手工维护。
4. 执行 `wv dev --open` 或 `wv ide logs --open`。
5. 展示日志桥接、截图或截图对比能力。

### 6.2 为什么不建议现场演太多功能

- 小程序 IDE 现场环境不稳定，演示越长越容易失控。
- 听众记不住十个命令，但能记住一个“从开发到验收被串起来”的闭环。
- 论坛场景更适合讲“方法论 + 代表性能力”，不是做完整产品培训。

### 6.3 备选方案

如果现场网络、IDE 或端口环境不稳定，建议改成“静态截图 + 命令行流程图”：

- 一张自动路由 / 自动导入的配置示意图
- 一张分包与 shared strategy 示意图
- 一张日志桥接 / screenshot / compare / MCP 的闭环图

## 7. 高频问答

**Q1：它和 `uni-app`、`Taro` 这类框架最大的区别是什么？**

定位不同。`weapp-vite` 优先服务“仍想保留原生小程序模型”的团队，在原生基础上增强工程化；跨端框架更强调统一范式和多端复用。

**Q2：它为什么不直接做成跨端框架？**

因为很多团队的真实诉求并不是换框架，而是保留原生能力、降低研发成本。`weapp-vite` 选择先把原生小程序这条链路做深做完整。

**Q3：它最明显的收益出现在哪类项目？**

中大型原生项目里最明显，尤其是分包多、组件多、npm 依赖复杂、多人协作、需要自动化验收和 AI 参与研发的项目。

**Q4：老项目适合接入吗？**

适合。它本身就强调渐进迁移，保留原生目录结构，同时通过托管 TypeScript 支持文件和 CLI 降低改造成本。

**Q5：自动路由和自动导入组件，真的只是少写配置吗？**

不是。它们真正减少的是“长期维护漂移”。项目越大，这类漂移带来的返工和隐性故障越明显。

**Q6：分包能力为什么值得重点讲？**

因为真实小程序项目一复杂，首包体积、独立分包、共享依赖、样式共享、npm 落位都会变成持续治理问题，而不只是一次性配置问题。

**Q7：AI 协作为什么能算核心功能，而不是附属噱头？**

因为今天很多团队已经让 AI 参与研发了，但多数 AI 只能停留在“解释代码”。`weapp-vite` 的价值在于把日志、截图、对比和文档入口都接进来，让 AI 能参与真实调试和验收。

**Q8：Web runtime 是不是等于可以不打开 DevTools 了？**

不是。Web runtime 更适合预览、调试和兼容验证，不应该被表述成微信小程序真机或 DevTools 的生产等价替代。

**Q9：多平台能力现在应该怎么讲？**

可以讲成“已经给未来多平台输出留出能力边界”，但要明确它仍是实验特性，不要在论坛场合把它当成成熟的主卖点。

**Q10：怎么判断团队该不该试？**

如果团队最大的诉求是“一套代码跑所有端”，优先看跨端框架；如果团队要的是“原生不推倒重来，但研发方式必须升级”，就值得试。

| 对齐对象        | 要确认什么                                               | 原因                             |

### 9.1 最低彩排标准

1. 至少完整讲一遍，控制在 24 到 26 分钟。
2. 至少练一次开场和结尾，不要临场现想。

- 避免连续三页都在念功能名词，功能一定要绑定“它解决了什么痛点”。

## 10. 风险、开放问题与下一步

### 10.1 风险

- 如果讲太多功能罗列，听众会记不住产品定位。
- 如果把多平台、Web runtime、Vue SFC 讲太重，会稀释主线。
- 如果现场 Demo 依赖 DevTools 环境，稳定性风险高。

### 10.2 开放问题

- 论坛听众更偏原生开发者还是架构治理人群。
- 是否有足够稳定的现场环境演示 `wv ide logs` 或截图能力。
- 是否需要在分享里加入真实业务迁移案例。

### 10.3 下一步

1. 根据论坛听众画像，把 Slide 6 到 Slide 9 的篇幅再微调一次。
2. 决定是做 live demo 还是静态截图演示。
3. 按这份文档把 PPT 正式排版成 10 到 12 页版本。
