# Weapp-vite 功能总结与 PPT 提炼指南

## 1. 概述

`weapp-vite` 不是单一的“小程序构建工具”，而是一套围绕小程序研发流程搭建的现代化工具链。它尽量保持原生小程序的目录结构、语法习惯和运行时边界不变，同时把 Vite 生态、TypeScript、样式预处理、自动化构建、AI 协作和 DevTools 自动化能力整合进来。

对外表达时，可以把它概括成一句话：

> `weapp-vite` = 保留原生小程序开发方式，同时补齐现代工程化、调试、AI 协作与多场景交付能力的研发工具链。

## 2. 产品定位

### 2.1 它要解决什么问题

| 问题                    | 传统方案痛点                                               | weapp-vite 的解决方式                            |
| ----------------------- | ---------------------------------------------------------- | ------------------------------------------------ |
| 原生开发体验偏弱        | 类型、模块化、样式工程化、自动化能力不足                   | 接入 Vite/Rolldown、TypeScript、样式预处理与 CLI |
| 存量项目迁移成本高      | 换跨端框架意味着重写心智与业务结构                         | 尽量保留原生目录和语法，支持渐进迁移             |
| 分包、npm、组件维护复杂 | `usingComponents`、`miniprogram_npm`、共享依赖需要人工维护 | 自动导入组件、自动 npm 构建、分包与 chunk 策略   |
| 调试链路割裂            | 编辑器、终端、开发者工具来回切换                           | CLI、IDE 打开、日志桥接、截图、截图对比          |
| AI 难以理解真实项目     | 模型只会泛化回答，缺少项目内真实执行能力                   | `AGENTS.md`、本地文档、MCP、截图与日志工具接入   |

### 2.2 它不是什么

| 维度                | weapp-vite 的立场                                       |
| ------------------- | ------------------------------------------------------- |
| 跨端大一统框架      | 不是。它优先服务原生小程序研发模式                      |
| 纯构建器            | 不只是。它还覆盖脚手架、IDE 自动化、AI 协作、运行时配套 |
| 生产等价 Web 运行时 | 不是。Web runtime 更偏开发预览和兼容验证                |

## 3. 核心能力地图

### 3.1 工程化基础能力

| 能力                   | 说明                                                                            | 价值表达                             |
| ---------------------- | ------------------------------------------------------------------------------- | ------------------------------------ |
| 原生写法兼容           | 保持小程序原生语法和目录结构                                                    | 降低迁移与学习成本                   |
| Vite/Rolldown 构建体系 | 基于 Vite 配置体系和 Rolldown 打包能力                                          | 兼顾生态兼容与小程序定制优化         |
| TypeScript 托管支持    | 自动生成 `.weapp-vite/*` 支持文件和托管 tsconfig，思路接近 Nuxt 的 prepare 流程 | 让新项目和老项目都能渐进接入类型系统 |
| CSS 工程化             | 支持 PostCSS、Sass、Less 等样式能力                                             | 让样式体系更接近现代前端             |
| JSON 增强              | 支持 JSON 智能提示、TS/JS 生成 JSON                                             | 降低配置文件维护成本                 |
| 脚手架生成             | `create-weapp-vite`、`wv generate`、`wv init`                                   | 缩短项目初始化与页面组件搭建时间     |

### 3.2 小程序特色增强

| 能力                      | 说明                                                                                      | 价值表达                       |
| ------------------------- | ----------------------------------------------------------------------------------------- | ------------------------------ |
| 自动路由                  | 扫描主包/分包页面目录，生成路由清单与类型                                                 | 约定式页面管理，减少手工维护   |
| 自动导入组件              | 自动维护 `usingComponents`，支持本地组件和 UI 库 resolver                                 | 组件越多，收益越明显           |
| 页面 Layout / Route Rules | 借鉴 Nuxt `app/layouts` 思路，通过 `layouts`、`definePageMeta`、`routeRules` 统一页面外壳 | 提升页面结构复用能力           |
| 自动构建 npm              | 自动输出 `miniprogram_npm`，支持主包、分包、插件独立控制                                  | 降低 npm 落位和依赖治理成本    |
| 分包能力完整支持          | 支持普通分包、独立分包、分包样式、分包依赖策略                                            | 面向真实复杂小程序场景         |
| Chunk 策略优化            | 支持 `duplicate` 和 `hoist` 等共享策略                                                    | 在首开性能和包体复用之间做取舍 |
| Worker 构建               | 按 `app.json.workers` 自动编译 worker 入口                                                | 覆盖小程序多线程能力           |
| 插件开发支持              | 支持插件目录与插件侧 npm 构建                                                             | 适用于插件型业务和平台能力封装 |

### 3.3 开发、调试与交付能力

| 能力                                 | 说明                                                               | 价值表达                     |
| ------------------------------------ | ------------------------------------------------------------------ | ---------------------------- |
| `dev` / `build` / `analyze` / `open` | 覆盖开发、构建、分析、打开 IDE 主链路                              | 一套 CLI 统一研发操作        |
| 分包分析                             | `wv analyze` 输出分包/产物映射结果                                 | 更容易做包体与依赖治理       |
| DevTools 集成                        | 可透传 `preview`、`upload`、`config`、`screenshot`、`compare`      | 构建链路与 IDE 操作打通      |
| 日志桥接                             | `wv ide logs --open` 与 `forwardConsole` 把运行时日志回传终端      | 减少在多个工具之间切换       |
| 截图与截图对比                       | 支持运行时截图、基线图对比、视觉回归式验收                         | 更适合自动化验收和 AI 操作   |
| `prepare` 预热                       | 提前生成 `.weapp-vite` 支持文件，行为上接近 Nuxt 的 `nuxt prepare` | CI、编辑器和老项目接入更稳定 |

### 3.4 AI 协作能力

| 能力           | 说明                                                    | 价值表达                              |
| -------------- | ------------------------------------------------------- | ------------------------------------- |
| MCP 接入       | `wv mcp`、`wv mcp init/print/doctor`                    | 把项目真实能力暴露给 AI 客户端        |
| AI 意图路由    | 通过 `AGENTS.md` 约束截图、对比、日志等动作走原生命令   | 避免 AI 误用 Web 工具替代小程序运行时 |
| 本地随版本文档 | `node_modules/weapp-vite/dist/docs/*.md`                | AI 能优先读取当前安装版本说明         |
| AI 友好脚手架  | `create-weapp-vite` 默认生成 `AGENTS.md` 并引导技能安装 | 降低团队接入 AI 的门槛                |

### 3.5 进阶与实验能力

| 能力             | 说明                                                           | 价值表达                         |
| ---------------- | -------------------------------------------------------------- | -------------------------------- |
| Vue SFC 编译支持 | `weapp-vite` 负责 `.vue -> 小程序产物` 编译，`wevu` 负责运行时 | 让小程序可用现代 Vue SFC 开发    |
| Web runtime      | 支持浏览器侧预览、兼容验证与调试                               | 提供非 DevTools 的快速验证路径   |
| 多平台构建       | 支持 `weapp`、`alipay`、`tt`、`swan`、`jd`、`xhs`、`h5` 等目标 | 一套工程可面向多个小程序平台输出 |
| Lib 模式         | 用于构建小程序组件库或业务模块库                               | 支持团队沉淀可复用产物           |

## 4. 差异化价值提炼

### 4.1 最值得讲的五个卖点

1. **原生不推倒重来**：不强迫团队切换到重抽象跨端框架，保留原生小程序研发习惯。
2. **工程化能力补齐**：把 TypeScript、样式工程化、自动生成、模块化能力带进小程序开发，并吸收 Nuxt `prepare` 这类成熟工程化思路。
3. **对复杂小程序更友好**：分包、npm、共享 chunk、worker、插件，以及借鉴 Nuxt `layouts` 思路的页面壳组织能力，都面向真实项目难点。
4. **调试与验收闭环更完整**：不仅能构建，还能打开 IDE、转发日志、截图、截图对比、做分析。
5. **天然适配 AI 协作**：不只是“能让 AI 读代码”，而是让 AI 能调用小程序真实能力完成验收和排障。

## 6. 可直接放进 PPT 的文案

### 6.1 30 秒版本

`weapp-vite` 是一套面向原生小程序研发的现代化工具链。它不要求团队放弃原生目录和语法，而是在保留现有开发模式的前提下，引入 Vite 生态、TypeScript、样式工程化、自动路由、自动组件注册、分包优化、IDE 自动化和 AI 协作能力。它的目标不是替代原生，而是让原生小程序拥有更现代、更高效的研发体验。

### 6.2 90 秒版本

从功能上看，`weapp-vite` 已经不只是一个构建器，而是一整条小程序研发链路。前面有 `create-weapp-vite` 和 `wv generate` 负责初始化与生成，中间有 `dev`、`build`、`prepare`、`analyze`、`open` 负责日常开发构建，再往后还有分包分析、日志桥接、截图、截图对比、DevTools 自动化，以及面向 AI 的 MCP 接入。对于复杂项目，它提供自动路由、自动导入组件、分包与 chunk 策略、npm 落位、worker、插件与 lib 模式；对于未来演进，它还提供 Vue SFC、Web runtime 和多平台构建等能力。因此，`weapp-vite` 更适合被理解为“小程序现代研发工具链”，而不是单点构建工具。

## 7. 适用场景总结

| 场景                           | 是否适合               |
| ------------------------------ | ---------------------- |
| 原生小程序团队想升级工程化体验 | 非常适合               |
| 有存量项目，希望渐进迁移       | 非常适合               |
| 业务复杂，分包、npm、插件较多  | 非常适合               |
| 希望把 AI 接入真实开发流程     | 非常适合               |
| 需要一套代码强覆盖所有端       | 需要结合跨端框架再评估 |

## 8. 总结

如果要用一句更适合汇报的话来收尾，可以直接使用下面这段：

`weapp-vite` 的核心价值，不是把小程序“框架化”，而是把小程序研发“现代化”。它保留原生、增强工程化、覆盖复杂场景、打通调试闭环，并进一步把 AI 协作纳入标准研发流程。对于既重视原生能力、又希望提升研发效率的团队，它提供了一条成本更低、可渐进落地的升级路径。
